# Booking Service

Сервис бронирования с асинхронной обработкой через Celery.

## Основные функции

- REST API для управления бронированиями (создание, получение, список, отмена)
- Фоновая обработка через Celery с вероятностью сбоя 15%
- Идемпотентность задач
- Экспоненциальный retry с backoff для упавших задач
- Structured logging в JSON
- Rate limiting на создание бронирований (5 запросов в минуту)

## Стек технологий

- **API:** FastAPI
- **Фоновые задачи:** Celery + Redis
- **База данных:** PostgreSQL (в продакшене), SQLite (для тестов)
- **Миграции:** Alembic
- **Контейнеризация:** Docker, Docker Compose
- **Тестирование:** pytest
- **Линтеры:** flake8, black, isort

## Запуск

### Через Docker Compose (рекомендуется)

```bash
# Клонировать репозиторий
git clone <repo-url>
cd booking_service

# Скопировать переменные окружения
cp .env.example .env

# Запустить стек
docker-compose up --build
```

Сервис будет доступен по адресу: http://localhost:8000
Swagger UI: http://localhost:8000/docs

### Локальная разработка (без Docker)

```bash
# Создать виртуальное окружение
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate

# Установить зависимости
pip install -r requirements.txt

# Применить миграции (SQLite по умолчанию)
alembic upgrade head

# Запустить API
uvicorn app.main:app --reload

# В отдельном терминале запустить Celery worker (требуется Redis)
celery -A app.tasks worker --loglevel=info
```

## Запуск тестов

```bash
# Все тесты
pytest

# С подробным выводом
pytest -v

# Только API тесты
pytest tests/test_api.py

# Только тесты воркера
pytest tests/test_worker.py
```

Тесты используют SQLite и не требуют запущенного Docker.

## Принятые технические решения

### Почему Celery, а не TaskIQ?

Celery — зрелое и стабильное решение для фоновых задач, широко используется в продакшене. Для данного задания выбор Celery оправдан простотой интеграции с Redis и PostgreSQL, наличием встроенных механизмов retry и мониторинга.

### Идемпотентность

Задача `process_booking` проверяет статус бронирования перед обработкой. Если статус не `pending`, задача завершается без изменений. Это гарантирует, что повторный запуск не испортит данные.

### Retry с экспоненциальным backoff

При возникновении исключения задача повторяется с задержкой, удваивающейся после каждой попытки (60, 120, 240 секунд). Это снижает нагрузку на внешние сервисы при временных сбоях.

### Structured logging

Логи выводятся в формате JSON, что упрощает их агрегацию и анализ в системах вроде ELK или Datadog.

### Rate limiting

Ограничение 5 запросов в минуту на создание бронирований защищает API от случайного перегруза. Реализовано через библиотеку `slowapi`.

### Тесты

Тесты используют SQLite и моки для Celery и random, что позволяет запускать их изолированно, без внешних зависимостей.
