# Техническое задание: Сервис бронирования

## 1. Общее описание проекта

### 1.1. Назначение системы
Сервис бронирования предоставляет REST API для управления бронированиями с асинхронной обработкой через фоновые задачи.

### 1.2. Основные функции
- Создание бронирований
- Получение информации о бронированиях
- Список бронирований с фильтрацией и пагинацией
- Отмена бронирований
- Асинхронная обработка бронирований через Celery
- Имитация случайных сбоев (15% вероятность)

---

## 2. Функциональные требования

### 2.1. Управление бронированиями

#### 2.1.1. Создание бронирования
**Endpoint:** `POST /bookings`

**Требования:**
- Принимает данные: имя клиента, дата/время, тип услуги
- Создает запись со статусом `pending`
- Запускает фоновую задачу для обработки
- Возвращает созданное бронирование

**Валидация:**
- `name`: строка, обязательное поле
- `datetime`: валидная дата и время в формате ISO 8601
- `service_type`: строка, обязательное поле

#### 2.1.2. Получение бронирования
**Endpoint:** `GET /bookings/{booking_id}`

**Требования:**
- Возвращает полную информацию о бронировании по ID
- Возвращает HTTP 404 если бронирование не найдено

#### 2.1.3. Список бронирований
**Endpoint:** `GET /bookings`

**Требования:**
- Поддержка фильтрации по статусу (`pending`, `confirmed`, `failed`)
- Поддержка пагинации через параметры `skip` и `limit`
- Значения по умолчанию: `skip=0`, `limit=10`
- Максимальное значение `limit`: 100

#### 2.1.4. Отмена бронирования
**Endpoint:** `DELETE /bookings/{booking_id}`

**Требования:**
- Позволяет удалить только бронирования со статусом `pending`
- Возвращает HTTP 400 при попытке отменить подтвержденное или проваленное бронирование
- Возвращает HTTP 404 если бронирование не найдено

### 2.2. Фоновая обработка

#### 2.2.1. Обработка бронирования
**Требования:**
- Выполняется асинхронно через Celery
- Проверяет идемпотентность (не обрабатывает уже обработанные бронирования)
- С вероятностью 15% устанавливает статус `failed`
- В остальных случаях устанавливает статус `confirmed`
- Имитирует отправку уведомления (mock)
- Логирует все действия

## 4. Модель данных

### 4.1. Таблица `bookings`

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | Integer (PK) | Уникальный идентификатор |
| `name` | String (Indexed) | Имя клиента |
| `datetime` | DateTime | Дата и время бронирования |
| `service_type` | String | Тип услуги |
| `status` | Enum | Статус: pending, confirmed, failed |

### 4.2. Статусы бронирований
- `pending` - ожидает обработки
- `confirmed` - подтверждено
- `failed` - обработка не удалась

## 5. API Спецификация

### 5.1. Схемы данных

#### BookingCreate
```json
{
  "name": "string",
  "datetime": "2026-06-20T10:00:00",
  "service_type": "string"
}
```

#### BookingResponse
```json
{
  "id": 1,
  "name": "string",
  "datetime": "2026-06-20T10:00:00",
  "service_type": "string",
  "status": "pending"
}
```

### 5.2. Endpoints

| Метод | Путь | Описание | Коды ответов |
|-------|------|----------|--------------|
| POST | `/bookings` | Создать бронирование | 201, 422 |
| GET | `/bookings/{id}` | Получить бронирование | 200, 404 |
| GET | `/bookings` | Список бронирований | 200, 400 |
| DELETE | `/bookings/{id}` | Отменить бронирование | 204, 400, 404 |

## 6. Требования к инфраструктуре

### 6.1. Контейнеры
1. **db** - PostgreSQL 15
   - Порт: 5432
   - Пользователь: postgres
   - База данных: bookings

2. **redis** - Redis 7
   - Порт: 6379

3. **web** - FastAPI приложение
   - Порт: 8000
   - Автоматическое применение миграций при старте
   - Hot reload в режиме разработки

4. **worker** - Celery worker
   - Обработка фоновых задач
   - Подключение к Redis и PostgreSQL

```env

```

## 9. Инструкция по развертыванию

### 9.1. Локальная разработка
```bash
# Установка зависимостей
python -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# Инициализация Alembic
alembic init alembic
# Настройка alembic/env.py (см. документацию)
alembic revision --autogenerate -m "Initial migration"

# Запуск тестов
pytest

# Запуск API локально
uvicorn app.main:app --reload
```

### 9.2. Запуск через Docker Compose
```bash
# Копирование переменных окружения
cp .env.example .env

# Сборка и запуск контейнеров
docker-compose up --build

# Остановка
docker-compose down
```

- Нет конкуренции за одни и те же временные слоты
